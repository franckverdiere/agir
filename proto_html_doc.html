Généralités

Règles transverses pour la nouvelle application Java/Angular

Fraîcheur des données — le proxy Java

Dans la nouvelle application, un système de proxy a la responsabilité de charger les données du modèle Legacy. Toutes les classes n'ont pas les mêmes besoins de rechargement : chaque fiche de cette documentation porte donc, en haut de page, un badge de rafraîchissement indiquant comment le proxy doit traiter la classe.

Stable — pas de rechargement à chaque appel : un chargement initial (cache) suffit

Volatile — à rafraîchir systématiquement à chaque appel

Spécifique — modalités décrites au cas par cas, selon les cas particuliers de la fiche

Règle : les classes volatiles se requêtent avec un discriminant de date

Sauf cas particulier, une classe volatile ne doit pas être requêtée « en entier » : la requête doit porter un élément discriminant de date. Exemple : les commandes d'une journée, ou d'une plage de dates.

Certaines classes volatiles n'ont pas d'attribut de date — par exemple les activités/navires (Activity). En règle générale, elles doivent alors être requêtées par jointure depuis une classe datée : pour l'activité/navire, la jointure part du chantier/escale (Task), qui porte les dates.

Requêtes sans limite : la tolérance ne concerne que les classes volatiles non datées : pour elles seulement, une requête sans restriction reste possible — par exemple pour proposer l'ensemble des navires dans une liste — mais cela doit rester rare et motivé. Les classes volatiles datées (chantiers/escales, commandes, postes, affectations…) doivent systématiquement faire l'objet de requêtes avec contraintes de date, sans exception.

Activity

Table Oracle : ACTIVITIES  |  Serveur C++ : tcMrpsActivity  |  Client Rp : tcMrpsRpActivity  |  Formulaire : tcMrpsFormActivity

Description fonctionnelle

La classe Activity modélise à la fois les navires et les activités de manutention hors exploitation (maîtrise, maintenance, etc.). Dans l'interface Legacy, la fiche correspondante s'appelle d'ailleurs « Fiche des activités/navires ».

Un navire est donc une Activity avec l'attribut vessel = 1. Ce choix de modélisation fait qu'un navire ne possède pas de classe dédiée : ses caractéristiques physiques (length, beam) sont portées directement sur Activity.

Chaque Activity regroupe des Task, qui représentent ses escales (pour un navire) ou ses chantiers (pour une activité hors exploitation). Le type de l'activité est déterminé par la relation vers ActivityType, qui forme une hiérarchie arborescente (exploitation, maîtrise, maintenance…).

Note d'architecture : les attributs length et beam sont purement informatifs. Ils sont gérés directement par le formulaire via Ilog Inform et ne transitent pas par le serveur ni le mapping du client BD, car les algorithmes n'en ont pas besoin.

ActivityType

Table Oracle : ACTIVITY_TYPES  |  Serveur C++ : tcMrpsActivityType  |  Client Rp : tcMrpsRpActivityType  |  Formulaire : tcMrpsSqlFormActivityType

Description fonctionnelle

La classe ActivityType est le référentiel des types d'activité : la classification métier qui qualifie chaque Activity (exploitation, manutention, maintenance, absence…). C'est une donnée de paramétrage, gérée dans la fiche Types d'activité.

Le référentiel est hiérarchique (arborescence) : chaque type peut avoir un type parent (parentActivityType) et donc des sous-types. La fiche présente d'ailleurs ces types sous forme d'arbre. Un type sans parent est une racine.

Les deux libellés label (libellé court) et name (nom complet) ainsi que la priority sont les seuls champs obligatoires.

Conventions Legacy à connaître :
  • Un type dont le code externe vaut « 00 » désigne un type de congé / d'absence (méthode isVacation()). C'est une valeur « magique » utilisée par le code Legacy ; côté Java, mieux vaut ne pas reposer sur cette convention de chaîne mais prévoir un indicateur explicite.
  • Il existe un type racine « Exploitation » (repéré par le paramètre ACTIVITY_TYPE_EXPLOITATION). Le code Legacy teste si un type descend de cette racine (isSubTypeOfExploitation()) pour distinguer les activités d'exploitation (navires, manutention) des autres.
  • Héritage hiérarchique : plusieurs attributs (les champs analytiques, le caractère fictif) ne sont pas lus uniquement sur le type courant mais remontent vers le parent s'ils ne sont pas renseignés localement. Ces attributs sont marqués dépréciés ci-dessous (non repris côté Java).

Task

Table Oracle : TASKS  |  Serveur C++ : tcMrpsTask  |  Client Rp : tcMrpsRpTask  |  Formulaire : tcMrpsFormTask  |  Panneau Gantt : tcMrpsTaskGantt

Description fonctionnelle

La classe Task modélise un chantier / escale : l'unité de travail planifiée pour un navire (ou une activité hors exploitation) sur une zone de travail donnée, pour une période donnée.

Le panneau principal est le Planning des Chantiers / Escales (Gantt), où chaque barre représente la durée d'un chantier et peut être déplacée par drag & drop. La fiche Fiche Chantiers / Escales existe mais est peu utilisée, potentiellement dépréciée.

Chaque Task est rattachée à une Activity (le navire ou l'activité concernée) et à une zone de travail (WorkingArea, le poste à quai). Le type d'opération est défini par la relation vers TaskType (chargement, déchargement, shifting…).

Lors de la création d'un chantier, si aucune cargaison n'est spécifiée, le chantier hérite de la cargaison de l'Activity parente (logique dans createRpTaskC2S de tcMrpsRpActivity.cpp). Les seuls champs véritablement obligatoires sont activity et workingArea : si l'association échoue, un rollback est déclenché.

Attention — convention de dates dans le modèle Legacy : dans Agir Legacy, les intervalles de dates suivent la convention dite « semi-ouverte » : la date de début (startDate) est incluse dans la période, tandis que la date de fin (endDate) est exclue. Concrètement, un chantier dont les dates vont du 1er au 3 avril couvre les journées du 1er et du 2, mais pas le 3. Cette convention est celle du modèle de données Oracle et restera inchangée. En revanche, les nouvelles interfaces utilisateur (côté Java/Angular) pourront présenter les dates différemment à l'écran (par exemple en dates inclusives). Le développeur doit donc être conscient de cet écart entre le stockage et l'affichage.

StevedoringSet

Table Oracle : STEVEDORING_SETS  |  Serveur C++ : tcMrpsStevedoringSet  |  Client Rp : tcMrpsRpStevedoringSet  |  Formulaire : tcMrpsFormStevedoringSet

Description fonctionnelle

La classe StevedoringSet représente une « Configuration d'engin/équipe » (libellé Legacy : « Config. Engins / Equipe »). C'est un gabarit de moyens pour une opération type : elle définit les engins nécessaires (nombre par classe d'engin, via EQUIPMENT_QUANTITIES) et la composition de l'équipe (effectifs par spécialité, via STRENGTH_DEFINITIONS).

Elle est au point de départ de la chaîne commande → affectation : la commande référence une configuration (SS_ID), et c'est elle qui détermine les postes générés. Une Activity peut porter une configuration par défaut : dans le panneau des commandes, choisir cette activité sélectionne automatiquement sa configuration.

Une configuration est de conduite opérationnelle (driving = 1) ou non — typiquement de maintenance (driving = 0). La distinction est structurante : la conduite opérationnelle exige un certain nombre de conducteurs pour tenir la cadence, tandis qu'en maintenance le nombre de conducteurs dépend de l'opération (par exemple un seul conducteur pour des déplacements d'engins ou des tests).

Note d'architecture : STRENGTH_DEFINITIONS et SS_GRADE_SUMS sont plus ou moins redondantes (composition par spécialité vs totaux par catégorie). Elles servent à générer les effectifs à la création de la commande. Pour connaître les spécialités réellement commandées (par exemple à l'initialisation d'un shift), interroger les postes de la commande, pas la configuration.

Grade

Table Oracle : GRADES  |  Serveur C++ : tcMrpsGrade  |  Client Rp : tcMrpsRpGrade  |  Formulaire : tcMrpsFormGrade

Description fonctionnelle

La classe Grade est le référentiel des « Catégories de personnel » (terminologie de l'interface Legacy). Une catégorie classe les agents (HumanResource) selon leur statut ou métier ; chaque agent est rattaché à une et une seule catégorie.

La catégorie structure aussi les spécialités : chaque Skill est rattachée à une catégorie. C'est un maillon des contrôles d'affectation : lors de l'affectation d'un agent à un poste, la compatibilité entre la catégorie de l'agent et celle exigée est vérifiée (violation « Cet agent n'est pas de la catégorie » dans le Gantt d'affectation).

La catégorie sert enfin d'axe d'agrégation : sommes de postes commandés par catégorie (par commande, par configuration d'engin/équipe), compteurs et quotas d'équité entre agents d'une même catégorie.

Note d'architecture : une catégorie référencée (agents, compteurs, sommes par commande) ne peut pas être supprimée (canBeDeleted).

Skill

Table Oracle : SKILLS  |  Serveur C++ : tcMrpsSkill  |  Client Rp : tcMrpsRpSkill  |  Formulaire : tcMrpsFormSkill

Description fonctionnelle

La classe Skill est le référentiel des « Spécialités » (terminologie de l'interface Legacy). Une spécialité décrit une compétence exercée par les agents : conduite d'un type d'engin, rôle dans une équipe, qualification particulière. Chaque spécialité est rattachée à une catégorie de personnel (Grade), rattachement obligatoire dans la fiche.

La spécialité est au cœur de la chaîne commande → affectation : chaque poste commandé exige une spécialité, et les contrôles d'affectation vérifient que l'agent proposé la pratique (violations « Aucune spécialité de l'individu n'est compatible » dans le Gantt d'affectation). Les niveaux de pratique des agents sont portés par l'entité PracticeLevel (table PRACTICE_LEVELS, hors périmètre de cette fiche).

Une spécialité a une durée de vie contrôlée selon son type : une spécialité « Simple » est toujours valide ; une spécialité « Entretenue par la pratique » ou « Entretenue par le renouvellement » expire au bout de periodOfValidity jours, avec une pré-alerte à periodOfAlert jours (panneau « Alertes spécialités » ; violations « période d'alerte » / « date de validité dépassée » lors des affectations).

Order

Table Oracle : ORDERS  |  Serveur C++ : tcMrpsOrder  |  Client Rp : tcMrpsRpOrder  |  Panneau : tcMrpsOrderSheet (carnet de commandes)

Description fonctionnelle

La classe Order représente une « Commande » de ressources pour un chantier / escale (Task). C'est l'acte par lequel l'exploitation commande, pour une zone de travail, un horaire et une configuration d'engin/équipe donnés, les moyens humains et matériels nécessaires à une opération. La commande génère les postes (Post) à pourvoir, qui seront ensuite affectés nominativement (Allocation). Le panneau principal est le carnet de « Commandes ».

La commande porte deux couples de dates, reflet de la distinction décrite sur la fiche WorkingPeriod : les dates de commande (orderStartDate / orderEndDate) correspondent à l'horaire commercial — l'horaire de location des engins (par exemple les portiques) ; les dates de travail (workStartDate / workEndDate) correspondent à l'horaire du personnel, plus large car il inclut l'équipement, la préparation, la mise au repos (du matériel) et le déséquipement.

La commande suit un cycle de vie (orderStatus) jalonné par un workflow de signatures : chaque passage d'état (planification, pointage, validation, clôture) est signé par un utilisateur habilité (fonctions ORDER_SIGN_*), l'historique étant conservé dans la table ORDERS_SIGNATURES.

Repris côté Java — le cycle de vie de la commande et son workflow de signatures sont conservés dans la nouvelle application.
Convention sur les intervalles de dates : comme partout dans Agir, les deux couples de dates sont des intervalles où la date de début est incluse et la date de fin est exclue.

Fonctionnalité « Démarrage de shift » — l'écran de démarrage présente les commandes du jour et de la plage horaire en cours, regroupées par escale. La requête ci-dessous détermine quelles commandes charger.

Requête Commandes du shift

Objectif : charger les commandes à afficher au démarrage du shift — celles du jour et de la (des) plage(s) horaire(s) courante(s) — regroupées par escale.

Démarche :

  • 1. Partir des plages horaires courantes — le résultat de la requête « Plages horaires courantes » (fiche AllottedSlot) donne les plages à considérer pour le centre d'activité courant.
  • 2. Retenir les commandes de ces plages — l'horaire de travail est porté directement par la commande (WPERIOD_ID) : retenir les commandes dont l'horaire de travail appartient à une des plages courantes (lien WorkingPeriod ↔ AllottedSlot via SlotDefinition), en se restreignant au jour courant sur les dates de travail (workStartDate / workEndDate). Aucun filtre sur le statut de la commande.
  • 3. Regrouper par escale et qualifier la commande — la commande porte son chantier / escale (TAS_ID) ; joindre l'Activity pour le nom du navire et son type d'activité, qui renseigne la nature de la commande (exploitation, assistance au sol, encadrement de conduite…). Joindre aussi la zone de travail (AREA_ID) pour le quai et, pour l'affichage, la configuration commandée (SS_ID), la cargaison et l'agent (CUSTOMER_ID).

Points de vigilance : (a) un même engin sur une même escale peut porter plusieurs horaires de travail successifs (shift principal puis prolongation) : ce sont des commandes distinctes, à rapprocher pour l'affichage « horaires successifs » de l'écran ; (b) les deux couples de dates suivent la convention semi-ouverte (début inclus, fin exclue).

Table Oracle : POSTS  |  Serveur C++ : tcMrpsPost  |  Client Rp : tcMrpsRpPost  |  Panneau : Gantt d'affectation (tcMrpsAllocationGantt)

Description fonctionnelle

La classe Post représente un « Poste » : une place de travail à pourvoir, générée par une commande. Chaque poste exige une spécialité et se déroule sur une plage horaire et un horaire de travail donnés. Convention sur les intervalles de dates : la date de début est incluse, la date de fin est exclue.

Fonctionnalité « Démarrage de shift » — le panneau de démarrage se construit en deux temps : d'abord sa structure (ce qui est commandé — requête ci-dessous), ensuite sa décoration avec ce qui est déjà affecté (requête « Affectations planifiées du shift », fiche Allocation). Requête Structure du panneau de démarrage

Objectif : construire le squelette du panneau uniquement à partir des données de commande — commandes, postes, configuration d'engin/équipe — indépendamment de toute affectation.

Démarche :

  • 1. Qualifier chaque commande — pour chaque commande retenue (cf. requête « Commandes du shift », fiche Order), sa nature est donnée par le type d'activité : exploitation (navire), assistance au sol, encadrement de conduite… C'est elle qui détermine dans quel bloc du panneau la commande s'affiche.
  • 2. Créer les emplacements d'engins — la configuration d'engin/équipe de la commande (SS_ID → StevedoringSet) donne le nombre d'engins commandés par classe d'engin (table EQUIPMENT_QUANTITIES : SS_ID + EC1_ID + QUANTITY) : un emplacement par engin commandé, même si aucun engin n'est encore affecté.
  • 3. Créer les places d'agents — les postes de la commande (POSTS.ORD_ID) sont les cases à pourvoir, chacune avec sa spécialité (SKILL_ID) et son horaire de travail (WPERIOD_ID), à grouper par horaires successifs (shift principal, prolongation).

La structure est entièrement déterminée par commande + configuration + postes. Les engins et agents déjà affectés viennent ensuite la « décorer » — voir la requête « Affectations planifiées du shift ».

Allocation

Table Oracle : ALLOCATIONS  |  Serveur C++ : tcMrpsAllocation  |  Client Rp : tcMrpsRpAllocation  |  Panneau : Gantt d'affectation (tcMrpsAllocationGantt, « Affectations »)

Description fonctionnelle

La classe Allocation représente une « Affectation » : le placement nominatif d'un agent (HumanResource) sur un poste. C'est une entité relationnelle à clé composite (POST_ID + HR_ID) : le poste est la demande issue de la commande, l'affectation est la réponse nominative — au plus une par poste. Le panneau principal est le Gantt d'« Affectations ».

L'affectation porte un cycle de vie (status) qui trace la communication avec l'agent : planifiée, puis publiée (portée à la connaissance de l'agent, date en pubDate), consultée (l'agent en a pris connaissance, date en consultDate), gelée ou consolidée.

Les contrôles de règles (catégorie, spécialité, validité, repos…) s'appliquent au moment de l'affectation. Une affectation peut être forcée malgré une violation : le motif est alors tracé dans reasonOfForcing.

Note d'architecture : les colonnes « terrain » (GROUND_SKILL_ID, GROUND_DELETION, GROUND_DELETOR) appartiennent au circuit Legacy des rectifications terrain (« Voir rectif. terrain ») alimentant la saisie du réalisé. Ce circuit n'est pas repris côté Java : attributs dépréciés.

Requête Affectations planifiées du shift

Objectif : décorer la structure du panneau avec les affectations issues de la planification Agir — les engins affectés aux commandes et les agents affectés aux postes — sachant qu'elles peuvent être complètes, partielles ou absentes.

Démarche :

  • 1. Partir des postes de la structure — les postes chargés par la requête « Structure du panneau de démarrage » (fiche Post).
  • 2. Des emplacements peuvent rester vides — l'engin sera choisi pendant le shift.

User

Table Oracle : USERS  |  Serveur C++ : tcMrpsUser  |  Client Rp : tcMrpsRpUser  |  Formulaire : UserForm

Description fonctionnelle

La classe User représente un utilisateur nommé d'Agir Legacy. Chaque utilisateur est identifié par un login (colonne LABEL) et porte un nom complet d'affichage (colonne NAME).

Un utilisateur est rattaché à un et un seul Profile (relation PROF_ID). C'est ce profil qui détermine ce que l'utilisateur a le droit de faire : l'utilisateur lui-même ne porte aucun droit en propre, il hérite de tous les droits définis sur son profil.

Cette logique d'administration restera à reproduire dans la nouvelle UI selon sa propre logique d'habilitation.

Profile

Table Oracle : PROFILES  |  Serveur C++ : tcMrpsProfile  |  Client Rp : tcMrpsRpProfile  |  Formulaire : ProfileSqlForm

Description fonctionnelle

La classe Profile est le pivot de la gestion des habilitations d'Agir. Un profil regroupe un jeu de droits sur les différentes fonctions applicatives, et un ou plusieurs User y sont rattachés. Tous les utilisateurs d'un même profil ont exactement les mêmes droits.

Concrètement, les droits sont matérialisés par les enregistrements Right : un Right est un triplet (Profile, Function, niveau d'accès) qui indique pour ce profil et cette Function, si l'accès est interdit, en lecture seule, ou en édition.

Le profil porte aussi un drapeau canUseAgir qui contrôle s'il a tout simplement le droit de se connecter à Agir Legacy. Un profil avec canUseAgir = 0 existe en base mais ne pourra ouvrir aucune session.

Autorisations par centre d'activité (repris côté Java) — au-delà des droits globaux, un profil peut accorder un droit (profil, fonction) sur un centre d'activité précis, via la table PROFILE_DEPENDENCES (clé composite PROFILE_ID + FUNCTION_ID + ACTIVITY_CENTER_ID — voir la fiche ProfileDependence). Cette notion d'autorisation par périmètre est conservée dans la nouvelle architecture.

Mécanisme Legacy « sensibilité poste » (non repris) — dans Agir Legacy, ces autorisations par centre étaient couplées à une restriction par machine : quand un profil a computerSensitive = 1, ses droits étaient filtrés selon la machine de connexion (table COMPUTER_ACCESS, hors périmètre), et PROFILE_DEPENDENCES servait à élargir l'autorisation « quelle que soit la machine ». Côté Java, seule la partie machine est abandonnée : COMPUTER_SENSITIVE reste déprécié, mais les enregistrements PROFILE_DEPENDENCES sont, eux, repris (lus directement comme « ce profil a ce droit sur ce centre »). Logique d'origine : RightsManagement::Profile::isAllowedTo*.

Le profil porte également la liste des types d'absence qu'il est autorisé à voir/gérer, via la table relationnelle VACATION_TYPES_OF_PROFILES (clé composite PROFILE_ID + VACATION_TYPE_ID). Right

Table Oracle : RIGHTS  |  Serveur C++ : tcMrpsRight  |  Client Rp : tcMrpsRpRight

Description fonctionnelle

La classe Right matérialise un droit accordé par un profil sur une fonction. C'est une entité relationnelle pure : sa clé primaire est le couple (PROF_ID, FUNC_ID), et son seul attribut métier est mask, le niveau d'accès.

Un profil et une fonction sont donc liés par au plus un enregistrement Right. L'absence d'enregistrement pour un couple (profil, fonction) équivaut à un droit nul (aucun accès, comme mask = 0).

La valeur de mask est un entier qui prend l'une de trois valeurs :

  • 0 — aucun accès (constante csNothing)
  • 1 — lecture seule (constante csRead)
  • 2 — lecture + édition (constante csEdit ; couvre allowedToRead et allowedToEdit)

Les deux méthodes du modèle Legacy allowedToRead() et allowedToEdit() reposent sur cette valeur : allowedToEdit est vrai uniquement si mask = 2 ; allowedToRead est vrai si mask = 1 ou 2. L'édition implique donc la lecture.

Note d'architecture : la colonne Oracle est typée Double dans le .dbd, mais le serveur C++ l'expose en int. Seuls les entiers 0, 1, 2 sont utilisés. Le nom historique « mask » suggère un masque de bits, mais ce n'est pas un bit field : c'est une simple énumération.

Function

Table Oracle : FUNCTIONS  |  Serveur C++ : tcMrpsFunction

Description fonctionnelle

La classe Function est le référentiel des fonctions applicatives sur lesquelles porte la gestion des habilitations. Chaque écran, panneau ou action sensible d'Agir est rattaché à une fonction identifiée par un libellé court (ex : USERADMIN pour l'administration des utilisateurs).

C'est un référentiel statique : la liste des fonctions est codée en dur dans l'application (constantes C++) et alimentée à l'installation. Il n'existe pas d'écran d'administration permettant d'ajouter ou de supprimer une fonction — ajouter une fonction implique de modifier le code et de livrer une nouvelle version.

Une fonction prend du sens en étant croisée avec un profil via Right : c'est ce croisement qui détermine le niveau d'accès effectif. La même fonction peut ainsi être en édition pour un profil, en lecture seule pour un autre, et interdite pour un troisième.

Note d'architecture : la liste des fonctions reflète la structure des écrans Legacy (administration utilisateurs, gestion des chantiers, validation des pointages, etc.). Côté Java, la nouvelle UI devra définir son propre référentiel de fonctions/actions : le mapping 1-pour-1 avec les fonctions Legacy n'a pas de sens, mais le mécanisme général (un libellé identifie une zone fonctionnelle, croisée avec un profil pour produire un niveau d'accès) peut être conservé.

ProfileDependence

Table Oracle : PROFILE_DEPENDENCES  |  Serveur C++ : ProfileDependence  |  Client Rp : RpProfileDependence

Description fonctionnelle

ProfileDependence est une entité relationnelle pure (clé composite (PROFILE_ID, FUNCTION_ID, ACTIVITY_CENTER_ID), aucun attribut métier propre) qui exprime une autorisation par centre d'activité : chaque enregistrement déclare un triplet (profil, fonction, activity center) pour lequel le Profile obtient le droit d'accès à la Function sur le périmètre de cet ActivityCenter.

Repris côté Java. Cette entité conserve sa sémantique actuelle — un droit (profil, fonction) accordé sur un centre d'activité donné. Le concept d'autorisation par périmètre est conservé dans la nouvelle architecture.

Découplage du mécanisme « machine ». Dans Agir Legacy, ProfileDependence était associé à la restriction par poste de travail : quand un Profile a computerSensitive = 1, ses droits étaient filtrés selon l'activity center atteignable par la machine de connexion (table COMPUTER_ACCESS), et ProfileDependence servait à élargir l'autorisation à un activity center « quelle que soit la machine ». Côté Java, seul ProfileDependence est repris : la partie machine (computerSensitive, Computer, COMPUTER_ACCESS) reste Legacy et n'est pas reprise. ProfileDependence se lit donc directement comme « ce profil détient ce droit sur ce centre d'activité ».

Logique Legacy d'origine : implémentée dans RightsManagement::Profile::isAllowedTo* (un droit était effectif si la machine donnait accès à l'activity center ou si un ProfileDependence existait pour le triplet). Auparavant exprimé en « working area », renommé en « activity center » en 07/2008 (cf. historique de ProfileDependence.h).

AllottedSlot

Table Oracle : ALLOTTED_SLOTS  |  Serveur C++ : tcMrpsAllottedSlot  |  Client Rp : tcMrpsRpAllottedSlot  |  Formulaire : tcMrpsSqlFormAllottedSlot

Description fonctionnelle

AllottedSlot (libellé Legacy : « Plage horaire ») désigne une plage horaire large utilisée en planification — typiquement Matin, Après-midi, Nuit, Shift étendu, etc. Elle ne représente pas un horaire de travail précis : c'est un regroupement qui sert de cadre temporel pour l'affichage (lignes de Gantt) et pour les choix par défaut dans l'allocation.

Une plage horaire est associée à plusieurs WorkingPeriod (horaires de travail précis) via la table relationnelle SlotDefinition, qui porte un attribut rank définissant des priorités (horaire par défaut de la plage, plage par défaut de l'horaire — voir la fiche SlotDefinition pour le détail).

Une plage horaire est par ailleurs rattachée à un ou plusieurs centres d'activité (ActivityCenter) via la table SLOTS_OF_ACTIVITY_CENTERS : cette relation N:N détermine dans quels périmètres d'exploitation la plage est disponible.

Deux types de plages existent :

  • Plage normale : associée à un ou plusieurs WorkingPeriod et utilisée pour la planification effective.
  • Plage fictive (fictitious = 1, libellé Legacy « Intervalle seul ») : pas d'horaire de travail associé, n'apparaît pas dans le panneau de filtre. Utilisée pour le paramétrage des règles d'effectif (SupportRule — fiche à venir).

Les attributs displayingBeginTime / displayingEndTime définissent la fenêtre d'affichage de la plage dans les Gantt (par exemple « ce slot s'affiche de 06:00 à 14:00 »).

À propos des 8 bornes (startTimeLowerBound1/2, startTimeUpperBound1/2, endTimeLowerBound1/2, endTimeUpperBound1/2) : ces colonnes définissent des bornes de tolérance utilisées pour valider les heures saisies sur les pointages et commandes rattachés à la plage. Les paires « _1 » et « _2 » correspondent à deux sous-périodes (typiquement avant et après pause). Elles ne sont pas saisies directement dans la fiche utilisateur ; le formulaire les initialise par défaut puis elles peuvent être ajustées par les algorithmes.

Convention de dates/heures : les heures de cette entité sont des heures de la journée (type Time, pas Date), à interpréter dans le fuseau du terminal. Les intervalles suivent la convention semi-ouverte d'Agir Legacy : la borne de début est incluse, la borne de fin est exclue.

Requêtes

Fonctionnalité « Démarrage de shift » — écran (côté Java/Angular) qui travaille dans le contexte d'un centre d'activité courant et d'un moment donné (l'heure courante). Pour s'afficher correctement, il a besoin de savoir quelle plage horaire considérer à cet instant.
Attention au vocabulaire : un AllottedSlot n'est pas un shift. C'est une plage horaire (Matin, Après-midi, Nuit…). La requête ci-dessous sert à déterminer la plage horaire applicable, pas à lister des shifts.

Requête Plages horaires courantes

Objectif : déterminer les plages horaires à considérer pour le centre d'activité courant, en fonction de l'heure qu'il est. Le résultat attendu est l'ensemble des plages horaires (AllottedSlot) compatibles avec le moment présent — il n'y en a souvent qu'une, mais il peut y en avoir plusieurs.

Démarche :

  • 1. Restreindre au centre d'activité courant — ne considérer que les AllottedSlot rattachés au centre d'activité courant via la table SLOTS_OF_ACTIVITY_CENTERS (relation N:N). On obtient ainsi l'ensemble des plages horaires définies pour ce périmètre (typiquement Matin / Après-midi / Nuit).
  • 2. Retenir toutes les plages compatibles avec l'heure courante — parmi ces plages, garder toutes celles dont la tranche horaire contient l'heure qu'il est : l'heure courante doit tomber entre l'heure de début et l'heure de fin de la plage (displayingBeginTime / displayingEndTime). Le plus souvent une seule plage convient, mais plusieurs peuvent se chevaucher et être retenues ensemble.

Points de vigilance : (a) une plage de nuit peut chevaucher minuit (heure de fin < heure de début) — le test de compatibilité doit gérer ce passage de jour ; (b) les bornes de tolérance de la plage (startTime…Bound / endTime…Bound) peuvent affiner la fenêtre retenue si l'on veut tenir compte des marges autour des heures théoriques ; (c) les plages fictives (fictitious = 1, « Intervalle seul ») sont à écarter, elles ne correspondent pas à une tranche de travail réelle.

WorkingPeriod

Table Oracle : WORKING_PERIODS  |  Serveur C++ : tcMrpsWorkingPeriod  |  Client Rp : tcMrpsRpWorkingPeriod  |  Formulaire : tcMrpsSqlFormWorkingPeriod

Description fonctionnelle

WorkingPeriod (libellé Legacy : « Horaire de travail ») représente un horaire précis de travail avec ses paramètres : heure de début, durée, temps de préparation, temps de mise au repos, pause, etc. Par exemple : « 06/14 » = travail de 6h00 à 14h00 avec ses spécificités.

Un WorkingPeriod porte systématiquement deux horaires imbriqués qui ne désignent pas la même chose et n'ont pas le même usage :

  • L'horaire commercial = startTime + duration. Ce sont les heures pendant lesquelles les engins sont loués et en opération : c'est ce qui est facturé au client et ce qui structure l'exploitation portuaire (ex : 6h00 → 14h00).
  • L'horaire de travail = horaire commercial élargi de preparationDuration au début et de settingBackInOrderDuration à la fin. C'est l'horaire effectif des agents, qui arrivent avant l'opération commerciale (préparation, prise de poste, briefing) et repartent après (mise au repos, rangement, fermeture). Pour l'exemple ci-dessus avec 30 min de préparation et 15 min de mise au repos, l'horaire de travail va de 5h30 à 14h15.

Cette distinction est essentielle : les algorithmes d'allocation et les pointages raisonnent sur l'horaire de travail (présence effective de l'agent), tandis que la facturation et l'affichage commercial raisonnent sur l'horaire commercial.

Un WorkingPeriod peut être basique (isBasicWP = 1) ou composé : un horaire composé est défini comme la séquence de deux autres WorkingPeriod, via les relations firstSubWorkingPeriod et secondSubWorkingPeriod. Typiquement, un horaire « Journée complète » peut être composé de « Matin » + « Après-midi ». La fiche Legacy a un toggle « Composé » qui bascule entre les deux modes.

Un WorkingPeriod est rattaché à une ou plusieurs AllottedSlot (plages horaires) via la table SlotDefinition. Le rank de la SlotDefinition détermine si la plage est la plage par défaut de cet horaire ou non. Un horaire est aussi associé à des zones de travail (WORKING_PERIODS_OF_AREAS), des centres d'activité (WPERIODS_OF_ACTIVITY_CENTERS), des règles d'allocation et des contextes (tables relationnelles).

Convention de dates/heures : les heures sont des heures de la journée (type Time) ou des durées (typées Time elles aussi : 02:30:00 signifie 2 heures et 30 minutes). Les intervalles suivent la convention semi-ouverte Legacy.

SlotDefinition

Table Oracle : SLOT_DEFINITIONS  |  Serveur C++ : tcMrpsSlotDefinition  |  Client Rp : tcMrpsRpSlotDefinition

Description fonctionnelle

SlotDefinition est l'entité relationnelle qui relie un AllottedSlot (plage horaire) à un WorkingPeriod (horaire de travail précis). C'est une relation N:N non ferme : un même horaire peut faire partie de plusieurs plages, et une plage regroupe plusieurs horaires.

Au-delà du lien, SlotDefinition porte un attribut clé : le rank (rang), qui encode une priorité par défaut dans les deux directions de la relation.

Sémantique du rang — à connaître absolument :
  • rank = 0 — défaut mutuel : le WorkingPeriod est l'horaire par défaut de l'AllottedSlot, ET l'AllottedSlot est la plage par défaut du WorkingPeriod. Au plus une relation de rang 0 par couple.
  • rank = 1 — défaut unilatéral : l'AllottedSlot est la plage par défaut du WorkingPeriod (sens WP → AS), mais le WP n'est pas l'horaire par défaut de l'AS.
  • rank = 2 — relation ordinaire : le WP est rattaché à la plage, mais sans relation de défaut dans aucun sens. C'est la valeur des autres associations.

Cette mécanique permet aux algorithmes de répondre à deux questions distinctes :

  • « Quelle est la plage par défaut de ce WorkingPeriod ? » → chercher la SlotDefinition de ce WP dont le rang vaut 0 ou 1 (la plus petite valeur). Cette logique correspond à la méthode getMasterSlot() côté serveur.
  • « Quel est l'horaire par défaut de cette AllottedSlot ? » → chercher la SlotDefinition de cet AS dont le rang vaut 0.

Note d'architecture : la table est une association pure (composite SLOT_ID + WPERIOD_ID en PK + l'attribut RANK). Côté Java, conserver cette structure permet de garder la sémantique des défauts bidirectionnels. Si la nouvelle UI n'a pas besoin de la double notion de défaut, elle peut se contenter de stocker une simple appartenance et un drapeau « est par défaut ».
